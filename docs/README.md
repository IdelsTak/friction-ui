# Friction — Problem Discovery System of Record

Most tools tell you what to build.  
Friction answers a simpler, harder question:

**What pain is real, recurring, and provable?**

If you’re scanning Reddit for tech pain points, Friction is built to turn scattered complaints into **traceable evidence** and **measurable metrics**—without losing context or inventing conclusions.

## The Pain

When you research problems on Reddit, you usually hit these issues:

- **Insight decay**: your summaries overwrite the original context.
- **Keyword bias**: you miss conceptually similar complaints.
- **Manual overload**: tagging and synthesis don’t scale.
- **No proof chain**: it’s hard to justify *why* a pain point is real.

## What Friction Does

Friction is a **system of record** for unresolved challenges:

1. **Ingests** public discourse (Reddit first).
2. **Extracts observations** with full provenance.
3. **Clusters** recurring complaints into a single friction.
4. **Computes metrics** like prevalence, intensity, persistence, and trend.
5. **Surfaces evidence** with links back to original sources.

No recommendations. No opinions. Just measurable, auditable pain.

## Why It’s Different

- **Immutable evidence**: every metric links back to raw sources.
- **Event-driven**: scale ingestion, clustering, and projections independently.
- **Metric-first**: pain points are comparable across topics and time.
- **Developer-first**: API and read models, not just dashboards.

## Core Concept

Friction is built around a simple promise:

> *“If a pain point can’t be traced back to real evidence, it doesn’t count.”*

## Target Use Case (Today)

I’m researching tech pain points on Reddit and need to:

- identify recurring challenges
- cluster related complaints
- quantify how severe and persistent they are
- keep provenance intact so I can verify everything

Friction is designed to make that workflow precise and repeatable.
